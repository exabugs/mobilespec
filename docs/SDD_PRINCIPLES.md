# Structure-Driven Development (SDD)

## 構造駆動開発 原則

SDD（Structure-Driven Development）は、
アプリケーションを「実装」ではなく「構造」で管理する開発手法である。

---

## なぜ SDD が必要か

従来の開発では：

- 実装コードが正とされる
- 仕様書は古くなる
- AI は都度指示に依存する
- フレームワーク変更が困難

SDD では：

- 構造（Structure）を唯一の正とする
- 実装は生成物とみなす
- 契約違反は CI で検知する
- AI は構造生成の補助者とする

---

## SDD の三層構造

### 1. API 構造

- OpenAPI が唯一の API 契約

### 2. UI 構造

- L2: 画面遷移構造
- L3: 画面構成構造
- L4: 状態・データ契約構造

### 3. 補助構造

- i18n（キー参照のみ）
- Policy（権限制御）
- Design Tokens

---

## 実装の扱い

実装コードは：

- 再生成可能
- 破棄可能
- キャッシュである

---

## AI の位置づけ

AI は：

- 構造作成の補助
- 実装生成の補助

だが、

- 構造の正ではない
- 契約検知の主体ではない

---

## SDD の目的

- フレームワーク非依存
- 移植可能
- AI と共存可能
- トークン効率最大化
